## Result

On this 10-task fixed A/B run, `lightweight_memory` did not improve accuracy and substantially increased cost. The harmful-memory evaluator surfaced a real risk signal: all memory-labeled harmful events occurred on failed tasks 3 and 10, each with `harmful_rate = 1.0`.

This does not prove memory caused the failures; the labeler is heuristic and based on overlap/quality signals. But it does distinguish "memory injected and task failed with strong overlap" from "memory injected but neutral", which is the behavior this change was meant to expose.

## Memory DB Outcome

The clean `lightweight_memory` run ended with:

- Strategic memories: 19
- Operational memories: 16
- Failure patterns: 9
- Stored feedback counts across selected memories: helpful 3, neutral 21, harmful 6

This confirms the new failure-trajectory path is active: failed tasks generated failure patterns instead of being skipped.

## Interpretation

The core instrumentation works:

- Injected memories are tracked with `memory_id`.
- Per-task `memory_events` and `memory_feedback` are written.
- Helpful/neutral/harmful/stale/unsupported labels are aggregated.
- Failure trajectories produce failure-pattern memories.
- Harmful rate is now measurable for outer-loop selection.

But the current `lightweight_memory` policy is not yet good enough:

- Long-term and short-term memory add large token overhead.
- Online learning within the run increases cost quickly.
- Harmful labels appeared on failed tasks without an accuracy gain.
- Task 1 is the clearest cost regression: same correctness, `+222,611` tokens.

## Caveats

- This is a fixed-task A/B run, not a strict held-out train/test study.
- `lightweight_memory` was allowed to learn online from earlier tasks in the same 1-10 run.
- Web/search results and API retries are nondeterministic.
- Crawl/Jina quota issues appeared in logs, which can affect both systems.
- The harmful label is a risk signal, not causal proof.

## Recommendation

Do not claim quality improvement from this run. The safe conclusion is:

> The new evaluation harness works and exposes memory risk, but `lightweight_memory` currently trades equal accuracy for much higher token/time cost and a nontrivial harmful-memory signal.

Before a larger 20-50 task evaluation, tighten:

- long-term retrieval gating,
- short-term memory extraction budget,
- memory injection frequency,
- harmful/stale/unsupported suppression,
- held-out train/test protocol.
